# CI — GitHub Actions on a Self-Hosted Runner

## Overview

`.github/workflows/tests.yml` runs the backend suite (`Unit`, `Feature`, `Procedures`) on every push
and pull request to `main`, then a separate `browser` job (Pest 4 + Playwright) that waits for it to
finish. Both jobs run on `[self-hosted, linux, homelab, tailnet]`, not a GitHub-hosted runner — this
app's tests need to reach 3 real Tailscale-only database servers (`docs/03-db-architecture.md`), which
a GitHub-hosted runner has no path to without exposing those DBs publicly or handing Tailscale
credentials to GitHub Secrets. Both are worse than hosting the runner inside the same tailnet the
databases already live on.

The runner itself — the LXC container it runs in, its threat model, and its scoped Vault token for
reading DB credentials — is provisioned and documented as part of the homelab, not this repo:
[taufiq's homelab repo](https://github.com/tttaufiqqq/oracle-db-linux-proxmox)'s
[`docs/09-github-actions-runner/actions-runner-setup.md`](https://github.com/tttaufiqqq/oracle-db-linux-proxmox/blob/main/docs/09-github-actions-runner/actions-runner-setup.md).
This doc only covers the workflow file's own decisions and the one real bug found running it for the
first time.

## Why two jobs, not one

`browser` declares `needs: backend` so it never runs concurrently with it — the runner CT has a single
CPU core, so two `artisan`/`npm` processes at once would contend for the same core rather than actually
run faster. Both jobs also share a `concurrency: group: tests-shared-db` with
`cancel-in-progress: false`: the 5 databases each job resets via `db:fresh-all` are real shared
MariaDB/MySQL/PostgreSQL servers, not per-run containers, so two runs resetting them at the same time
would stomp on each other, and cancelling a run mid-reset would leave tables half-dropped.

## Incident: `shivammathur/setup-php@v2` fails on every run

The workflow originally used `shivammathur/setup-php@v2` to install PHP. The very first time this
workflow actually ran on real hardware (the workflow file existed in a prior commit but was never
pushed until 2026-07-19), every single job failed at that step with:

```
ENOENT: no such file or directory, open '/home/run-gh-runner/actions-runner/_work/_actions/shivammathur/setup-php/v2/src/scripts/linux.sh'
```

`/home/run-gh-runner/...` doesn't exist anywhere on the runner — the real path is
`/home/linux-gh-runner/...`. The action's own execution log showed the child Node process exiting in
**60 milliseconds**, far too fast to have actually attempted any PHP install logic — a strong signal
this was failing during path resolution, before real work started, not a flaky download.

### Root cause

Reproduced the exact call manually over SSH (matching the runner's own env vars and working directory)
to get a full picture, then grepped the actual compiled bundle
(`~/actions-runner/_work/_actions/shivammathur/setup-php/v2/dist/index.js`, action version `2.37.2`)
for the code building this path:

```js
const r = e + await g.scriptExtension(e);              // e = "linux"; r = "linux.sh"
const n = path.join(__dirname, "../src/scripts", r);    // n = ".../src/scripts/linux.sh"  — correct
const i = n.replace(e, "run");                          // bug
```

`String.prototype.replace(string, replacement)` only replaces the **first** match. `n` is the full
absolute path, not just the filename, and this runner's own directory is named `linux-gh-runner` — the
substring `"linux"` (`e`, the platform name being substituted) appears there, inside the hostname,
*before* it appears in the trailing `linux.sh` filename the code actually meant to target. So the
replace corrupts:

```
/home/linux-gh-runner/actions-runner/.../src/scripts/linux.sh
       ^^^^^ first match — wrongly replaced
```

into a path that was never real:

```
/home/run-gh-runner/actions-runner/.../src/scripts/linux.sh
```

This is a genuine bug in the action's own compiled code, not a corrupted cache, a download race, or
anything specific to this repo's app — it happens to be triggered here specifically because the
homelab named this runner `linux-gh-runner`, and the OS platform string `setup-php` substitutes is
also `"linux"`. Confirmed by direct reproduction (not just log-reading): running the exact same
`node externals/node24/bin/node .../dist/index.js` invocation by hand, with `GITHUB_ACTION_PATH` and
every other env var the real runner passes set explicitly, still produced the identical wrong path —
ruling out anything env-specific to a single job run.

### Fix

Dropped the action entirely rather than trying to pin a different tag/SHA of it (the bug is in a
fundamental, likely long-standing code path, not a recent regression, so an untested older/newer ref
could easily carry the same defect). `.github/workflows/tests.yml` now installs PHP directly via `apt`
instead — the same PPA-free approach `infrastructure/ansible/playbooks/app-server.yml` already uses in
production, since Ubuntu 24.04 ships PHP 8.3 natively:

```yaml
- name: Install PHP 8.3 and required extensions
  run: |
    sudo apt-get update
    sudo apt-get install -y php8.3-cli php8.3-mbstring php8.3-bcmath php8.3-curl \
      php8.3-xml php8.3-zip php8.3-mysql php8.3-pgsql

- name: Install Composer
  run: |
    command -v composer >/dev/null 2>&1 && exit 0
    curl -sS https://getcomposer.org/installer | php
    sudo mv composer.phar /usr/local/bin/composer
    sudo chmod +x /usr/local/bin/composer
```

This also bumps the pinned PHP version from 8.2 to 8.3, matching what production actually runs
(`composer.json` only requires `^8.2`, so this is a strictly compatible move, not a version bump that
needed its own review).

**Why not rename the runner instead?** Renaming `linux-gh-runner` would dodge this one collision, but
is a bigger, separate infrastructure change (re-registering with GitHub, updating the homelab's DNS
alias for it) that belongs in the homelab repo if it's ever done, and wouldn't rule out the same class
of bug resurfacing from some other future path/hostname collision. Removing the buggy dependency
entirely is the more durable fix.

## Incident 2: `migrate` fails with "could not find driver" for `sqlite`

With the `setup-php` step fixed, the very next run got further than ever before — `db:fresh-all`
dropped tables on all 5 real connections successfully — then failed at `migrate` itself:

```
could not find driver (Connection: sqlite, SQL: select exists (select 1 from "main".sqlite_master
where name = 'migrations' and type = 'table') as "exists")
```

### Why sqlite is involved at all

Every model and migration in this app pins one of the 5 real connections explicitly
(`docs/03-db-architecture.md`) — sqlite is never used for actual data. But Laravel's migration
**repository** (the bookkeeping that tracks which migrations have already run) always operates against
`config('database.default')`, regardless of which connection an individual migration file targets,
unless `migrate` is given an explicit `--database` flag (it isn't, here — see
`app/Console/Commands/FreshAllDatabases.php`). `.env.testing.example` deliberately sets
`DB_CONNECTION=sqlite` so that bookkeeping step never depends on any of the 5 real Tailscale databases
being reachable — a reasonable and intentional design choice, just one this CI runner wasn't actually
equipped to satisfy yet.

### Fix

Two things were missing, both now added to `.github/workflows/tests.yml`:

1. **The `pdo_sqlite` extension** — `php8.3-sqlite3` added to the same `apt-get install` step that
   installs PHP.
2. **The database file itself** — `.env.testing.example` points `DB_DATABASE` at
   `database/testing.sqlite`, which is gitignored (it's a local artifact, not app data). A fresh CI
   checkout never has it, so a `mkdir -p database && touch database/testing.sqlite` step runs
   alongside `cp .env.testing.example .env.testing`.

Neither of these could have surfaced with the old `shivammathur/setup-php@v2` step in place — CI never
got far enough to reach `migrate` before Incident 1's fix landed.

## Incident 3: `GD extension is not installed`

With both prior fixes in, the next run got to **359 passed, 5 failed** — every failure was
`LogicException: GD extension is not installed` from
`Illuminate\Http\Testing\FileFactory::image()`, used by the animal/report photo upload tests
(`AnimalCrudTest`, `ReportSubmissionTest`) to fake an uploaded image in-memory.

The original `shivammathur/setup-php@v2` step's `extensions:` input never listed `gd` either, so this
was always going to fail the moment CI got far enough to reach these tests — not a regression from
switching to `apt`, just the next gap in line. Fixed by adding `php8.3-gd` to the same `apt-get install`
step.

Three incidents in, in order of what CI actually reached: `setup-php` itself (Incident 1) → the sqlite
migration-repository connection (Incident 2) → a missing extension for image-upload tests (Incident 3).
Each fix unblocked the next failure rather than fixing everything at once — expected, since this
workflow had genuinely never run to completion before.

## Incident 4: a real test bug, not a CI/environment gap

Unlike the first three, this one isn't about the runner or its PHP install — the `backend` job passed
in full, and the `browser` job's `Tests\Browser\ForcedPasswordChangeTest` failed all 3 of its tests with
`RoleDoesNotExist: There is no role named 'adopter' for guard 'web'`.

`tests/Concerns/SeedsMinimalDomain.php`'s `makeAdopter()`/`makeAdmin()`/etc. helpers all call
`seedRoles()` before assigning a role — every other Browser test that needs a role-bearing user goes
through one of those. `ForcedPasswordChangeTest::makeForcedResetUser()` is the one place that instead
builds a `User` by hand and calls `$user->assignRole('adopter')` directly, and it's a plain global
function (not a method on the test case), so it never had access to `$this->seedRoles()` to begin with.
This test suite had apparently never run to a clean, completed state in this environment before, so the
gap had never been exercised.

Fixed directly in the test file: `Role::firstOrCreate(['name' => 'adopter'])` before the assignment,
mirroring what `seedRoles()` does for that one role.
