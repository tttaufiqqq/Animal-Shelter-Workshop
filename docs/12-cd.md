# CD â€” Automatic Deploy to the Fleet on a Self-Hosted Runner

## Overview

`.github/workflows/deploy.yml` fires automatically on `workflow_run` once `.github/workflows/tests.yml`
(`docs/11-ci.md`) finishes successfully on `main`. A push to `main` is the only human action in the
whole path: green tests â†’ the fleet converges to that exact commit â†’ smoke tests â†’ automatic rollback
to the last known-good commit if anything fails. There is no approval gate and no required manual
`ansible-playbook` step. `workflow_dispatch` exists only as an escape hatch (forced re-run, or a manual
rollback to an older SHA), never as something the normal path depends on.

Like `tests.yml`, this runs on `[self-hosted, linux, homelab, tailnet]` (`linux-gh-runner`, CT 111) â€”
the databases and app-server are Tailscale-only, and the runner already lives inside that tailnet. See
the homelab repo's `docs/09-github-actions-runner/actions-runner-setup.md` for the runner itself; this
doc covers the deploy design.

## Why a `provision` / `deploy` tag split

`playbooks/app-server.yml` does two very different jobs: one-time-ish host setup (apt packages, UFW,
TLS/certbot, Vault Agent installation, the scheduler's systemd unit) and per-push application deploy
(git pull, Composer, npm build, `.env` render, migrations, caches). Every task carries a `tags:
[provision]` or `tags: [deploy]`. CD runs `--tags deploy` unless the pushed commit actually touched
`infrastructure/ansible/` (see routing below), in which case it runs the full playbook instead. A CD
run **never** carries the `provision` tag alone and never seeds â€” `Seed databases (first deployment
only)` and the `.provisioned` marker tasks are tagged `provision` specifically so a deploy-tagged run
can't reach them even if the `provisioned_flag` check were ever wrong.

The 5 database playbooks (`linux-mysql.yml`, `linux-mysql-2.yml`, `linux-mariadb.yml`,
`linux-mariadb-2.yml`, `linux-postgres.yml`) aren't tag-split â€” they're small and almost entirely
"provision-shaped" already (packages, UFW, the one-time database/user creation). CD runs them in full,
but only when routing says to (below).

## Path-based routing â€” what actually runs

The `plan` job diffs the pushed commit against whatever SHA is recorded as *currently live* on
app-server (`storage/.last-good-deploy`, written only after a deploy's smoke tests pass â€” see below),
not against the previous push. That distinction matters: it makes a batch of several pushes, or a
retry after a failed deploy, route the same way a single clean push would.

| Changed paths | `deploy-db` runs | `deploy-app` runs |
|---|---|---|
| App code only | no | `--tags deploy` |
| `infrastructure/ansible/playbooks/app-server.yml`, `playbooks/tasks/**`, `templates/**`, `group_vars/**`, `inventory-ip.yml` | no | full playbook |
| `infrastructure/ansible/playbooks/linux-*.yml` | yes (`--limit databases`) | `--tags deploy` |
| No `.last-good-deploy` marker exists yet | yes | full playbook |
| `workflow_dispatch` with `force_full: true` | yes | full playbook |

`deploy-app` always runs (app-server is deployed on every push); `deploy-db` only runs when routing
says a DB playbook changed. If the `git diff` against the recorded last-good SHA fails for any reason
(e.g. that SHA no longer exists in history after a rebase), routing forces full playbooks rather than
guessing.

## Why `deploy-db` halts instead of rolling back

`deploy-app` has automatic rollback (below); `deploy-db` does not. There's no sane automatic reversal
of an apt package install or a UFW rule change on a live database host â€” "undo" isn't a well-defined
operation there the way redeploying a previous git SHA is for app-server. So `deploy-db` simply fails
loud and stops the pipeline before `deploy-app` ever touches app-server (`deploy-app`'s `if:` requires
`deploy-db` to have succeeded or been skipped â€” never failed). This is the one place in this pipeline
where "fully automated" means "automatically halts," not "automatically undoes."

## The `workflow_run` SHA gotcha

On a `workflow_run` trigger, `actions/checkout@v4` defaults to the repository's default branch tip â€”
**not** the commit that was actually tested. Every checkout step in this workflow pins
`ref: ${{ github.event.workflow_run.head_sha || inputs.ref }}` explicitly. Getting this wrong would
mean CD silently deploys whatever `main` happens to be at deploy time, which could already differ from
what CI just certified if another push landed in the gap.

## Rollback and its migration caveat

On any `deploy-app` failure (ansible run or smoke test), the `rollback` job redeploys
`storage/.last-good-deploy`'s SHA with `--tags deploy`, then re-runs the same smoke tests. If there's
no last-good SHA to roll back to (the very first CD deploy failed), a separate `no-rollback-target` job
fires instead and fails loud with a clear summary rather than silently doing nothing.

**Rollback restores code only.** Laravel migrations are forward-only, and MySQL/MariaDB DDL is
non-transactional â€” if a failed deploy got as far as `php artisan migrate` before failing later (e.g.
at the HTTP smoke test), the schema can end up ahead of, or inconsistent with, the rolled-back code.
Automatic `migrate:rollback` is deliberately never attempted: reversing a schema change automatically,
on a database whose true state after a failed run isn't fully known, risks destroying data instead of
fixing anything. The `rollback` job's summary always includes this caveat and asks for a manual
`php artisan migrate:status` check â€” it doesn't try to detect whether migration was actually the
failing step, since that detection would itself be one more thing to get wrong.

## Why the health check asserts on stdout, not exit code

The post-deploy health check runs `vault agent -config=/etc/vault-agent/agent-verify.hcl`, which chains
`migrate:status && db:refresh-status && about --only=environment`. `db:refresh-status`
(`app/Console/Commands/RefreshDatabaseStatus.php`) always returns `Command::SUCCESS`, even when every
one of the 5 DB connections is down â€” it's a status-reporting command, not a health-gate one. The
smoke test greps the captured output for the literal string `5/5 online` instead of trusting the exit
code. This is exactly the class of check that caught a real production incident before: the Laravel
scheduler was once silently running with blanked DB passwords after a Vault Agent rollout, and
`db:refresh-status` kept exiting 0 the whole time while reporting `0/5 online`.

## Secrets â€” `secret/asw-cd`, no long-lived credentials in GitHub

Following the runner's existing rule ("no DB credentials in GitHub Secrets, ever" â€”
`actions-runner-setup.md`), CD's own credentials live in Vault too, at `secret/asw-cd`: a dedicated
CD-only SSH private key (`ssh-keygen -t ed25519`, distributed to `authorized_keys` on all 6 fleet
hosts, independently revocable from the lab-wide admin key), the shared become/root password, and the
`asw-deploy` / `asw-app-server-agent` AppRole id/secret pairs `group_vars/all.yml` needs. The existing
`gh-runner` Vault policy was extended with one read-only path (`secret/data/asw-cd`) â€” no token
re-issue needed, policy changes apply to the runner's existing token immediately. Every job step reads
the key fresh from Vault into `$RUNNER_TEMP`, masks every value with `::add-mask::`, and shreds the key
file in an `if: always()` cleanup step.

## Incident: `login_unix_socket` broke on all 4 MySQL/MariaDB hosts

Found live during the very first bootstrap run for this CD setup â€” unrelated to anything CD-specific,
but it blocked the run. `CLAUDE.md`'s Server Topology note records that all 5 DB servers' root/postgres
passwords were unified to one value on 2026-07-20. On MySQL/MariaDB, setting a password via
`ALTER USER ... IDENTIFIED BY` switches that user's auth plugin from `unix_socket`/`auth_socket` (which
never checks a password) to `mysql_native_password`. `linux-mysql.yml`, `linux-mysql-2.yml`,
`linux-mariadb.yml`, and `linux-mariadb-2.yml` all connected via `login_unix_socket:` â€” a connection
method that stopped being valid for root on all 4 hosts the moment that unification landed, and nothing
had re-run those playbooks' DB-creation tasks since to notice. Fixed by adding a `mysql_root_password`
group var (`qwertY@1612`, matching what `CLAUDE.md` already documents in plaintext for this
homelab-with-no-real-data) and switching all 4 playbooks' `mysql_db`/`mysql_user` tasks to
`login_user`/`login_password`.

## Incident: `become_user: postgres` crashed with a Windows-style ACL error

Also found during bootstrap, also pre-existing and unrelated to CD's design: `linux-postgres.yml`'s
`Create workshop_2 database` task (which runs as `become_user: postgres`) crashed with
`chmod: invalid mode: 'A+user:postgres:rx:allow'`. This is the same bug `app-server.yml` already has a
fix and a comment for â€” Ansible's privilege de-escalation needs `setfacl` to hand an unprivileged user
access to its temp files, and falls back to a non-Linux ACL dialect that fails outright on Ubuntu when
`acl` isn't installed. `app-server.yml` installs `acl` for its own `become_user: www-data` tasks;
`linux-postgres.yml` never got the same fix. Added the identical `Install acl` task there.

## Incident: the frontend build task never ran twice

Not something CD triggered directly, but a real, already-live bug found while auditing the deploy path
for a pinned-SHA rollback to even be meaningful: `Build frontend assets` (now split into `Install npm
dependencies` + `Build frontend assets`) was guarded by `creates:
public/build/manifest.json`. Once that file exists â€” which it has, since the very first deploy â€” every
subsequent `npm ci && npm run build` was silently skipped forever, regardless of whether the frontend
actually changed. Every Vue/Blade/CSS change since then shipped stale compiled assets. Fixed by
registering the git task (`register: app_repo`) and gating the build on `when: app_repo.changed`
instead of a `creates:` guard â€” this is also what makes rollback and no-op redeploys correctly skip the
multi-minute `npm ci` on app-server's single core when nothing actually moved.

Separately, that task's original form, `command: npm ci && npm run build`, was already latent-broken:
Ansible's `command:` module never invokes a shell, so `&&` was being passed to `npm` as a literal argv
token, not chaining two commands. It happened to look like it worked only because the `creates:` guard
above meant this had only ever executed once.

## Known follow-up: fork-PR workflow approval, unverified

This repository is public (`tttaufiqqq/Animal-Shelter-Workshop`), by deliberate choice â€” a portfolio
demo repo, not something with a private-by-default threat model. Combined with a self-hosted runner
that holds Vault credentials reaching every DB in the lab, GitHub's own guidance is that
`pull_request`-triggered workflows on self-hosted runners are a real risk (anyone can open a PR).
Whether this repo's "Require approval for first-time contributors" fork-PR gate is actually enabled
could not be checked from this environment (no `gh` CLI or API token available at the time this was
set up) â€” confirm it manually under **Settings â†’ Actions â†’ General â†’ Fork pull request workflows**
before treating this as closed.
